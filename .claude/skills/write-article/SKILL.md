# 書きたいことドリブン記事作成スキル

「書きたいこと」を起点に、対話で素材を深掘りしながら採用広報・社内広報記事のドラフトを生成するスキル。インタビュー形式ではなく、ユーザー自身が持つアイデアや断片的なメモから記事を組み立てる。

## 前提ファイル

このスキルは以下のファイルと連携する。すべてプロジェクトルートからの相対パスで参照すること。

- `.claude/skills/write-article/writing-guide.md` - 深掘り対話ガイドと質問パターン
- `.claude/skills/write-article/article-template.md` - 記事テンプレートと文章ルール
- `.claude/agents/write-article-drafter.md` - 記事生成サブエージェント

## ワークフロー

### Phase 1: 素材投入

AskUserQuestion ツールを使い、記事の出発点を収集する。

**質問1: 入力タイプの選択**

```
question: "記事の出発点を選んでください"
options:
  - label: "テーマ・キーワードがある"
    description: "書きたいテーマや伝えたいキーワードがある（例: 「チームの技術選定プロセス」「オンボーディング改善」）"
  - label: "断片メモがある"
    description: "箇条書きメモや思いつきの断片がある（例: 「・新しいCI/CD導入した ・開発速度2倍になった ・チームの反応よかった」）"
  - label: "両方ある"
    description: "テーマも断片メモもある"
```

**質問2: 素材の入力**

入力タイプに応じた質問を行う。

- 「テーマ・キーワードがある」場合:

```
question: "書きたいテーマやキーワードを教えてください。複数あれば全て書いてください"
```

- 「断片メモがある」場合:

```
question: "メモや思いつきをそのまま貼り付けてください。箇条書き、走り書き、どんな形式でも構いません"
```

- 「両方ある」場合: テーマとメモを順番に取得する

**質問3: 著者の確認**

```
question: "この記事の著者（書き手）は誰ですか？著者の一人称視点で記事を書きます"
options:
  - label: "自分が著者"
    description: "自分自身の一人称視点で記事を書く（名前・役職を教えてください）"
  - label: "別の人が著者"
    description: "記事の主役となる人物の視点で書く（名前・役職を教えてください）"
  - label: "著者なし（三人称）"
    description: "特定の著者を置かず、組織やチーム視点で書く"
```

取得した情報を内部的に保持する。「著者なし（三人称）」の場合は `author` を `null` とする。

**質問4: 記事の種類とターゲット**

```
question: "記事の種類とターゲットを教えてください"
options:
  - label: "採用広報（エンジニア向け）"
    description: "エンジニア候補者に自社の魅力を伝える記事"
  - label: "採用広報（非エンジニア向け）"
    description: "非エンジニア候補者に向けた組織・文化紹介の記事"
  - label: "社内広報"
    description: "社内メンバーに向けた取り組み紹介・ナレッジ共有の記事"
  - label: "テックブログ（採用広報兼）"
    description: "技術発信しつつ採用ブランディングも兼ねる記事"
```

**質問5: 記事の登場人物（任意）**

```
question: "著者以外に記事に登場する人物がいれば教えてください（名前、役職、チーム名など。いなければ「なし」で構いません）"
```

取得した情報を以下の形式で内部的に保持する。

```
context:
  input_type: {テーマ / 断片メモ / 両方}
  raw_material:
    theme: {テーマ・キーワード（あれば）}
    memo: {断片メモ（あれば）}
  author:
    name: {著者名}
    role: {役職}
    perspective: {一人称 / 三人称}
  article_type: {記事の種類}
  target_persona: {ターゲット}
  characters: {著者以外の登場人物（あれば）}
```

### Phase 2: 素材の深掘り対話

`.claude/skills/write-article/writing-guide.md` を読み込み、深掘りパターンを把握する。

#### 対話の進め方

1. Phase 1 で投入された素材を分析し、記事として不足している要素を洗い出す
2. 不足要素に対して AskUserQuestion で1問ずつ質問する
   - question に質問文を設定する
   - 回答の参考になる選択肢を2〜3個用意しつつ、「Other」からの自由入力を基本とする
3. 回答を受け取ったら、深掘りルールに照らして判定する
4. 深掘りが必要な場合は追加質問を行う（最大2回）
5. 次の質問を前の回答の内容を踏まえて動的に生成する
6. 合計3〜6問を目安に素材を充実させる

#### 素材分析の観点

投入された素材を以下の観点で分析し、不足している要素を特定する。

| 観点 | 不足の判定基準 | 深掘りの方向 |
|---|---|---|
| 具体性 | 固有名詞・数値・技術名がない | 具体的な名前や数値を引き出す |
| エピソード | 出来事や体験の記述がない | 印象的なエピソードを聞き出す |
| 背景・動機 | 「なぜ」「どう始まった」が不明 | 経緯や動機を掘り下げる |
| 成果・変化 | Before/After が見えない | 取り組みの前後でどう変わったかを聞く |
| 感情・実感 | 事実の羅列のみで温度感がない | やってみてどう感じたかを引き出す |
| メッセージ性 | 読者に何を伝えたいかが不明確 | 一番伝えたいことを言語化してもらう |

#### 深掘りの判定

writing-guide.md の深掘りルールに従う。

- 回答が30文字未満 → 具体的なエピソードを求める
- 抽象的な表現のみ → 場面や数値を含む具体例を求める
- 「なぜ」が抜けている → 動機や背景を掘り下げる
- 深掘りは同一質問で最大2回。それでも具体性が不足すれば次の質問へ進む

#### 素材データの蓄積

各質問と回答を以下の形式で蓄積する。

```
material_log:
  raw_input:
    theme: {テーマ}
    memo: {断片メモ}
  enriched:
    - aspect: {観点}
      question: {質問文}
      answer: {回答文}
      followups:
        - question: {深掘り質問}
          answer: {深掘り回答}
```

### Phase 3: 構成提案

深掘りが完了したら、記事の構成案を提示して確認を取る。

#### 構成案の生成

素材データ全体を俯瞰し、以下を設計する。

1. **仮タイトル**: 30文字以内、記事の核心を端的に表す
2. **TL;DR の方向性**: 1〜2文で概要を説明
3. **セクション構成**: 3〜4セクションの見出しと各セクションに割り当てる素材

#### ユーザーへの提示

構成案をテキストで提示し、AskUserQuestion で確認する。

```
question: "この構成で記事を作成してよいですか？"
options:
  - label: "この構成でOK"
    description: "このまま記事を生成する"
  - label: "構成を修正したい"
    description: "セクションの順番や内容を変更する"
  - label: "素材を追加したい"
    description: "Phase 2 に戻って追加の深掘りを行う"
```

- 「この構成でOK」→ Phase 4 へ進む
- 「構成を修正したい」→ 修正内容を聞いて構成を更新し、再度確認
- 「素材を追加したい」→ Phase 2 に戻って追加の深掘りを行う

### Phase 4: ドラフト生成

構成が確定したら、記事生成をサブエージェントに委任する。

#### サブエージェントの起動

Task ツールを使い、`write-article-drafter` サブエージェントを起動する。プロンプトに以下を含める。

```
以下の素材と構成案から採用広報/社内広報記事を作成してください。

## 記事の種類
{article_type}

## ターゲット
{target_persona}

## 著者情報
- 名前: {author.name}
- 役職: {author.role}
- 視点: {author.perspective}（一人称の場合は著者視点で「〜しました」「〜です」、三人称の場合は客観視点で記述）

## 登場人物
{characters（あれば）}

## 素材データ
### 元の入力
- テーマ: {theme}
- メモ: {memo}

### 深掘りで得た情報
{material_log の enriched 部分をテキストとして展開}

## 構成案
{Phase 3 で確定した構成案}

## 出力先
{プロジェクトルート}/articles/{YYYY-MM-DD}-{テーマのslug}.md

## 指示
1. `.claude/skills/write-article/article-template.md` を読み、テンプレートと文章ルールに従って記事を作成してください
2. 著者の視点に応じた文体で書いてください
   - 一人称: 著者の視点で「〜しました」「〜です」。著者自身の発言は地の文に溶け込ませ、他者の発言は「」引用で残す
   - 三人称: 客観的な視点で記述
3. 文体は「ですます」調で統一してください
4. 記事を出力先パスに書き出してください
5. `npx textlint {出力先パス}` を実行し、エラーがあれば修正してください
6. textlint エラーが0になるまで修正を繰り返してください
```

### Phase 5: レビュー・仕上げ

サブエージェントの完了後、以下を行う。

1. 生成された記事ファイルを Read ツールで読み込む
2. ユーザーに記事全文を提示する
3. AskUserQuestion で修正の要否を確認する

```
question: "記事のドラフトが完成しました。修正したい箇所はありますか？"
options:
  - label: "このままでOK"
    description: "修正不要、記事を確定する"
  - label: "部分的に修正したい"
    description: "特定のセクションや表現を修正する"
  - label: "全体的に書き直したい"
    description: "トーンや構成を含めて大幅に修正する"
```

- 「このままでOK」→ 完了メッセージを表示し、出力ファイルパスを伝える
- 「部分的に修正したい」→ 修正箇所を聞いて Edit ツールで対応
- 「全体的に書き直したい」→ フィードバックを聞いてサブエージェントを再起動
