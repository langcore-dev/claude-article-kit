# レビュー観点チェックリスト

## 1. 構成

### TL;DR
- [ ] 記事全体の要約として機能しているか（3〜4文）
- [ ] 読者が「この先を読みたい」と思えるフックがあるか
- [ ] 記事の結論や核心を先に示しているか

### セクション構成
- [ ] セクション数は3〜5が適切（多すぎず少なすぎず）
- [ ] セクション間の流れが自然で論理的か
- [ ] 各セクションに最低1つの「」引用があるか
- [ ] 各セクションが150〜300文字の目安に収まっているか
- [ ] セクション見出しが具体的で魅力的か（「技術について」のような汎用見出しはNG）

### タイトル
- [ ] 30文字以内か
- [ ] 記事の核心やハイライトを含んでいるか
- [ ] 読者の興味を引くフックがあるか

---

## 2. 文章ルール（textlint 準拠）

### 必須ルール
- [ ] 一文が100文字以内に収まっているか
- [ ] 文体が「ですます」調で統一されているか
- [ ] 助詞（の、に、が、を等）が同一文内で連続使用されていないか
- [ ] 冗長表現が排除されているか（「することができる」→「できる」）
- [ ] 全角/半角が適切か（技術用語は半角英数）

### 品質ルール
- [ ] 素材の言葉が「」引用として活用されているか
- [ ] 地の文と引用が交互に配置されリズムがあるか
- [ ] 読者がその場面をイメージできる具体性があるか

---

## 3. 視点の一貫性

### 一人称記事の場合
- [ ] 著者視点（「〜しました」「〜です」）で統一されているか
- [ ] 著者の発言が地の文に溶け込んでいるか（「」引用になっていないか）
- [ ] 他者の発言のみが「」引用になっているか
- [ ] 三人称の記述が混在していないか

### 三人称記事の場合
- [ ] 客観的な視点で統一されているか
- [ ] すべての発言が「」引用として扱われているか
- [ ] 一人称の記述が混在していないか

---

## 4. 具体性

- [ ] 固有名詞（人名、社名、プロダクト名、技術名）が適切に使われているか
- [ ] 数値データ（期間、件数、割合、倍率等）が含まれているか
- [ ] 抽象的な表現（「良い」「すごい」「頑張った」）が具体的なエピソードに置き換えられているか
- [ ] Before/After（取り組みの前後の変化）が示されているか

---

## 5. 訴求力

### 採用広報記事の場合
- [ ] 候補者が「ここで働きたい」と思える要素があるか
- [ ] 組織の文化や価値観が自然に伝わるか
- [ ] 技術的な面白さや成長機会が示されているか
- [ ] 具体的な仕事のイメージが湧くか

### 社内広報記事の場合
- [ ] 社内メンバーが「自分もやってみたい」と思える要素があるか
- [ ] 取り組みの背景と成果が明確か
- [ ] ナレッジとして再利用可能な学びが含まれているか

### メッセージ性
- [ ] 記事を通じて伝えたいメッセージが明確か
- [ ] 最終セクションで読者に残したい印象が示されているか

---

## 6. textlint チェック

```bash
npx textlint {ファイルパス}
```

- [ ] textlint エラーが0件であるか
- [ ] 自動修正で解決しないエラーがある場合は手動で対応済みか
